### PC Web 已知使用规则

- API 模式登录后首页必须是 `/tasks`，不能继续默认请求 `TASK-001`。普通账号带 Token 查询该演示任务可能得到 404，这是现有后端任务权限规则的结果。
- Mock 模式仍只用于前端离线开发；它不会模拟写操作状态机。
- 真实设备关联只能走 `POST /api/v1/devices/{device_id}/bind`；任务创建/编辑页面不直接提交 `device_id`，避免后端目前两套关联数据不同步。
- 任务创建、发出、到达、签收、拒收的网络重试必须使用幂等键。当前后端已实现创建、发出、到达、签收、拒收的幂等处理；小程序也应同样实现。

请完成：

1. 建立显式的设备响应 DTO，所有设备相关接口都不得返回 `device_secret`、`device_secret_hash`、签名材料或任何可用于伪造设备请求的值；已经登记的设备密钥应轮换。
2. 改为“未登记或未配置有效密钥的设备一律拒绝正式遥测/心跳”，或将旧匿名兼容接口放在明确受控、默认关闭的演示开关后；错误/缺失签名应为 401，重复 nonce 为 409。

5. 明确设备登记 UPSERT 的所有权、重新配网和密钥轮换规则，禁止任意发货方使用同一设备编号覆盖别人的名称、型号或密钥。

- 发货方绑定本人任务后，设备和任务详情显示同一个关联；解绑后设备为 `available`、`current_task_id=null` 且任务 `device_id=null`；

### H-03：硬件验收

1. 后端登记设备并绑定一个新建 `WD-...` 任务；
2. 设备连续上报，Web/小程序能看到温湿度、箱体、运动、电量和最后在线时间；
3. 暂停心跳并验证离线告警；
4. 有真实定位时，Web 任务地图显示后端历史中的坐标点数和相对轨迹，不生成虚假点；
5. 设置 `device_secret` 后，正确签名可上传，错误签名/重复 nonce 被拒绝；
6. 网络中断后恢复，数据不覆盖或重复破坏顺序。

## 5. 小程序成员：实际开发任务

### M-01：正式会话和请求底座

**现状**：小程序登录是本地角色选择，没有 Token；请求层不带 `Authorization`、401 清理、刷新 Token 或幂等键。

**负责人**：小程序公共底座成员
**优先级**：P0

完成：

- 接入 `/auth/login`、`/me`、`/permissions`、`/refresh`、`/logout`；
- 将 Token 放入合适的小程序本地会话存储，严禁打印或上传到日志；
- 所有正式 `/api/v1` 请求自动带 `Authorization: Bearer ...`；
- 401 清会话并回登录页；403/409 展示后端消息；
- 不依赖本地角色判断最终权限，角色仅控制可见菜单；
- 为创建、发出、到达、签收、拒收生成一次性 `Idempotency-Key`。

验收：Token 失效后页面有明确提示，重新登录后可继续刷新；不出现“本地显示已签收、后端实际未签收”。

### M-04：二维码交接（等待 B-05/B-07 明确）

流程：创建 handoff → 创建 QR token → 扫码 → 验证 → 接收方确认/拒绝。相关接口：

## 7. 联调统一验收链路

1. 后端部署可访问地址，提供受控管理员、发货、承运、接收四类测试账号及权限矩阵；更新 API 文档和 Mock。
