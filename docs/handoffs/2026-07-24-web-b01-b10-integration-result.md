## 当前结论

当前未发现阻塞 Web 管理端开发的后端接口问题。PC Web 已按最新契约接入任务准备、交接记录、动态二维码证据、真实文件上传、管理员聚合大屏和权限提示；本地 API 联调完成了以下路径：

1. 发出方登录后创建/查看任务。
2. 读取承运方与接收方候选目录，提交双方指派。
3. 提交装箱预检，后端将任务更新为 `pending_handoff`。
4. 在后端校验通过后发出任务，状态更新为 `in_transit`。
5. 创建 `sender_to_carrier` 交接记录并生成一次性动态交接码。
6. 上传真实 PNG 文件；浏览器计算 SHA-256，后端校验并保存；页面出现带 Token 的受控下载入口。
7. 普通发出方访问 `/screen` 被前端导航守卫提示并回到任务总览；管理员访问同一路由可读取 `/api/v1/dashboard/summary` 的真实聚合数据。

本地验证只使用忽略的本地数据库和临时联调账号；账号、密码、Token、上传存储文件均未写入版本库。

## 发现并已处理的问题

### 证据截图扩展名与真实内容不一致

`web-admin/docs/evidence/` 下现有若干文件名为 `.png`，但文件头为 JPEG。新后端会正确返回 `415`，避免伪装内容进入证据库。

Web 已增加上传前文件头校验：当浏览器 MIME/扩展名与 JPEG、PNG、PDF 实际签名不一致时，不发送请求并提示用户选择真实格式的文件。

后续如需把这些历史截图作为正式证据，请由素材维护人将扩展名改为 `.jpg`，或重新导出为真实 PNG；不要仅修改前端允许列表或关闭后端校验。

## 交给小程序端的联调事项

1. 使用正式登录接口及 Token，按账号角色展示任务。
2. 接收方扫描 `coldchain://handoff?token=...` 后，提取 `token` 调用 `POST /api/v1/qr-tokens/verify`；不要把二维码字符串当作 Token 原文直接提交。
3. 仅在验证成功后调用 `/handoffs/{handoff_id}/confirm`；若返回 `40932`，要求重新扫码，而不是本地绕过。
4. 签收前处理 `40933`，引导完成 `carrier_to_receiver` 交接。
5. 文件上传使用真实 JPEG/PNG/PDF 二进制和正确 MIME；不能把 JPEG 内容命名为 `.png`。
6. 人脸相关界面必须标注“模拟核验/人工复核”，不可宣称真实生物识别已完成。

## 交给硬件端与后端联调的事项

1. 硬件从旧上传接口迁移到正式遥测与心跳接口，并使用每台设备的 HMAC 签名；设备密钥只写入设备安全存储，不能交给 Web 或小程序持久化。
2. 用实际开发板完成一次“设备绑定 → 遥测/心跳 → 断网离线 → 恢复上报”的链路，确认大屏中的在线/离线数与告警一致。
3. 组长提供正式 HTTPS API 地址、管理员/三种业务角色测试账号和微信域名配置后，再执行远程三端联调。当前 `127.0.0.1` 仅用于后端所在电脑的本地开发。
4. 如需在 Web 显示历史附件列表，后端可在后续接口中补充“按任务/交接分页列出文件元数据”的受控查询；当前上传响应已提供即时下载，交接列表只返回附件数量，因此不阻塞本阶段交付。
